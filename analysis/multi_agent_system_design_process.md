## 💡 設計思考プロセス

### 第一段階: 問題分析
**課題**: 単一モデルでの処理では...
- トークン使用量が過大
- 処理速度が遅い
- コストが高い
- モデルの強みを活かせない

**洞察**: タスク特性に応じたモデル割り当てが必要
- 分析タスク → 高精度モデル
- 調査タスク → 高速モデル
- 数値計算 → 数値特化モデル
- 前処理 → 軽量モデル

### 第二段階: アーキテクチャ設計
**選択肢1**: 集中制御型
- ✅ 管理が容易
- ✅ 一貫性がある
- ❌ ボトルネックの可能性

**選択肢2**: 分散協調型
- ✅ スケーラブル
- ✅ 障害に強い
- ❌ 調整が複雑

**決定**: ハイブリッドアプローチ
- マスターエージェントで集中制御
- 専門エージェントで分散処理
- バランスを取った設計

### 第三段階: モデル選定
**Ollamaローカルモデル**:
- 🟢 qwen3:8b → 戦略推論（バランス良い）
- 🟢 codegemma:7b → 数値計算（数値に強い）
- 🟢 llama3.2:1b → 軽量処理（高速）

**OpenRouterフリーミアム**:
- 🟡 gemini-flash → 調査・要約（高速・低コスト）
- 🟡 llama-3.2-1b → 前処理（軽量）

**z.ai coding plan**:
- 🔴 高精度分析（高コスト故に限定利用）

### 第四段階: トークン最適化戦略
**問題**: マルチエージェントでもトークン使用量が増大する可能性

**解決策**:
1. **動的予算配分**: タスク重要度に応じたトークン割り当て
2. **メタデータ優先**: 全文検索前のメタデータ検索
3. **智能要約**: トークン制限内での効率的な要約
4. **キャッシュ活用**: 同一クエリの重複処理防止
